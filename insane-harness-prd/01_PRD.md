# insane-harness — PRD (Product Requirements Document)

> 생성일: 2026-07-07
> 생성 도구: Show Me The PRD

---

## 1. 제품 개요

### 한 줄 요약
Claude Code 워크스페이스의 세션 로그·스킬·구성을 진단해서, 그 워크스페이스에 필요한 **특성 하네스**(플러그인/훅/규칙/스킬)를 제안하고 승인받아 적용하며, 워크스페이스가 진화하면 하네스도 따라 진화시키는 메타 하네스 플러그인.

### 해결하는 문제
- 범용 하네스(fablize, insane-loop 등)는 깔려 있어도 "이 워크스페이스에서 무엇을 검증하고 언제 멈출지"는 워크스페이스마다 다르다. 지금은 그 특성 하네스(배포 체크리스트, 전용 스킬, 강제 훅)를 사람이 수작업으로 발견하고 만든다.
- 워크스페이스가 진화해도(작업 성격 변화, 새 반복 패턴) 하네스는 방치된다 — 안 쓰는 훅이 남고, 필요한 게이트는 늦게 생긴다.
- 어떤 플러그인이 이 워크스페이스에 맞는지 카탈로그를 뒤지는 일도 수동이다.

### 핵심 가치 (차별점)
공식 [Claude Code Setup 플러그인](https://claude.com/plugins/claude-code-setup)은 **코드베이스**를 읽고 추천만 하고 끝난다(읽기 전용). insane-harness는:
1. **행동을 본다** — 코드가 아니라 세션 로그(실제 작업 패턴·반복·에러)를 진단 원료로 쓴다.
2. **적용까지 한다** — 추천에서 끝나지 않고 승인 게이트를 거쳐 훅 머지·CLAUDE.md 주입·플러그인 설치 유도까지 수행한다.
3. **자가성장한다** — 적용 장부(ledger)를 남겨 재실행 시 증분 점검: 새 필요 제안 + 기존 하네스 유효성 재검증 + 안 쓰는 것 제거 제안.

## 2. 사용자

### 주요 사용자
- **누구**: Claude Code로 여러 성격의 워크스페이스(플러그인 개발, 책 집필, 실험, 리서치)를 오가는 파워 유저 / 바이브코더
- **상황**: 새 워크스페이스를 시작할 때, 또는 기존 워크스페이스에서 반복 작업·반복 실수가 쌓였다고 느낄 때
- **목표**: "이 폴더에서 일할 때 필요한 안전장치·절차·도구"를 자동으로 갖추고, 시간이 지나도 그 세팅이 낡지 않게 유지

### 사용자 시나리오
1. 사용자가 책 집필 워크스페이스에서 `/insane-harness`를 실행하고
2. 첫 셋업(성장 모드·주기 선택) 후 진단이 돌면 "집필 세션 71%, 스타일 결정 반복 감지, book-wrap 스킬 존재" 같은 프로필과 함께 하네스 제안 리스트가 뜨고
3. AskUserQuestion으로 항목별 승인하면 → CLAUDE.md에 집필 게이트 규칙 주입, 훅 등록, 부족한 스킬은 skillers-suda로 위임 생성, 맞는 플러그인은 설치 안내
4. 3주 뒤 재실행하면 "그 사이 세션 40개 분석: 교정 작업이 새로 늘었고, 지난번 훅 X는 최근 미사용" → 추가/제거 제안을 다시 승인받아 반영

## 3. 핵심 기능

| # | 기능 | 설명 | 우선순위 | 복잡도 |
|---|------|------|----------|--------|
| 1 | 셋업 | 첫 실행 감지 → 성장 모드(자가성장/스냅샷)·주기(수동/N일/N세션)·주기 동작(알림/자동 진단) AskUserQuestion → `.insane-harness/config.json` 저장 | P1 (MVP) | 간단 |
| 2 | 진단 (scan) | `bin/scan.py`가 세션 jsonl **전체 통계**(도구 히스토그램·반복 명령·에러/중단율·자주 만지는 파일·스킬 사용 빈도) + 워크스페이스 인벤토리(.claude/ 스킬·커맨드·에이전트·훅·CLAUDE.md) + 설치된 마켓플레이스 카탈로그를 구조화 JSON으로 산출. LLM은 최근/이상치 세션 소수만 심층 해석 | P1 (MVP) | 복잡 |
| 3 | 제안 (prescribe) | scan 결과를 하네스 카탈로그(시그널→처방 매핑표, `references/harness-catalog.md`)와 대조해 워크스페이스 프로필 + 하네스 제안 리스트 생성. 불확실한 판단은 AskUserQuestion으로 좁힘 | P1 (MVP) | 보통 |
| 4 | 적용 (apply) | 승인된 항목만 `bin/apply.py`로 실행: ① 플러그인 설치 안내(명령 제시) ② settings.json 훅 안전 머지 ③ CLAUDE.md 마커 구획 주입 ④ 맞춤 스킬은 skillers-suda 위임(미설치 시 설치 유도). 백업 후 실행, 실패 시 롤백 | P1 (MVP) | 복잡 |
| 5 | 장부·증분 (ledger) | 적용 내역을 `.insane-harness/ledger.json`에 기록. 자가성장 모드 재실행 시: 마지막 스캔 이후 diff 기반 새 제안 + 기존 항목 유효성 재검증(사용 흔적) + 스테일 항목 제거 제안. 제거도 승인 게이트 | P1 (MVP) | 보통 |
| 6 | 주기훅 (cadence) | 플러그인이 배포하는 SessionStart 훅 1개가 CWD의 config를 읽어 주기 도래 체크 → 설정에 따라 알림 또는 진단 시작 유도. config 없는 워크스페이스에선 침묵 | P1 (MVP) | 보통 |
| 7 | status / config 커맨드 | `/insane-harness:status`(적용 하네스·장부·다음 점검 조회), `/insane-harness:config`(모드·주기 변경) | P1 (MVP) | 간단 |

> ✅ 확정(2026-07-07): 주기훅은 **플러그인 hooks.json으로 전역 1개 배포 + 워크스페이스별 config로 분기**. 처방된 훅(하네스 내용물)은 각 워크스페이스 settings.json에 개별 적용. (A3)

> ✅ 확정(2026-07-07): 주기 도래 시 기본 동작은 **알림 → 사용자 승인 시 진단 시작**, 자동 진단 시작은 옵트인. (A4)

## 4. 사용자 흐름 (User Flow)

### 핵심 흐름 (메인 커맨드)
```
/insane-harness 실행
  → [첫 실행?] 셋업 카드(모드·주기) → config 저장
  → scan.py 실행 (결정론 진단)
  → LLM 해석: 워크스페이스 프로필 + 카탈로그 대조
  → [자가성장 모드] ledger diff → 증분 제안 (신규+재검증+제거)
  → 제안 리스트 제시 → AskUserQuestion 승인 (항목별)
  → apply.py 실행 (승인분만, 백업 → 적용 → 검증 → 실패 시 롤백)
  → ledger 갱신 + 결과 리포트
```

### 상세 흐름
1. **셋업**: 첫 실행에만. 성장 모드/주기/주기 동작 3문항 1콜. 이후 `/insane-harness:config`로 변경.
2. **진단**: scan.py는 세션 원문을 컨텍스트에 붓지 않고 통계 JSON만 반환. LLM은 최근 10세션 + 에러 최다 세션 3개만 서브에이전트로 심층 읽기.
3. **제안**: "이 워크스페이스는 무엇을 하는 곳" 프로필 1문단 + 제안 테이블(하네스 / 근거 시그널 / 종류 / 예상 효과). 확신 낮은 항목은 질문으로 확정.
4. **승인**: AskUserQuestion multiSelect로 항목 선택. "전부 건너뛰기"도 항상 옵션.
5. **적용**: 종류별 실행기. 플러그인 설치는 명령 안내(자동 설치 아님). CLAUDE.md는 `<!-- insane-harness:begin/end -->` 구획 안만 수정.
6. **장부**: 항목별 id·근거·적용 시각·상태 기록. 다음 실행의 비교 기준선.

## 5. 성공 기준

- [ ] gptaku_plugins 워크스페이스에서 진단→제안→승인→적용 E2E가 세션 1회 안에 완료된다
- [ ] scan.py가 성격이 다른 워크스페이스 3곳(플러그인 개발/책 집필/실험)에서 에러 없이 통계를 산출하고, 워크스페이스 프로필이 실제 성격과 일치한다
- [ ] 승인하지 않은 항목은 어떤 파일도 변경하지 않는다 (dry-run + 파일 diff로 검증)
- [ ] 자가성장 모드 재실행 시 이미 적용된 항목의 중복 제안이 0건이다
- [ ] 주기훅이 config 없는 워크스페이스에서 완전히 침묵하고, 주기 도래 워크스페이스에서만 1줄 출력한다
- [ ] 공식 Setup 플러그인이 못 하는 것 3가지(행동 분석·적용·성장)가 각각 실동작으로 시연된다

## 6. 안 만드는 것 (Out of Scope)

> 이 목록에 있는 건 Phase 1에서 만들지 않습니다.

- **스킬 생성 엔진** — skillers-suda에 위임. 이유: 기존 전문 플러그인 재사용, 책임 분리
- **플러그인 자동 설치** — 설치 명령 안내까지만. 이유: 설치는 사용자 신뢰 영역 + CLI 자동화 불확실성
- **외부 레지스트리 웹 검색 추천** — 설치된 마켓플레이스 카탈로그만. 이유: 미검증 플러그인 추천 위험 (Phase 3 후보)
- **하네스 효과 정량 측정** (적용 전후 지표 비교) — Phase 3. 이유: 장부 데이터가 쌓여야 가능
- **크로스-워크스페이스 종합 대시보드** — Phase 2+. 이유: 단일 워크스페이스 루프가 먼저 검증돼야 함
- **팀 공유/다중 머신 동기화** — 개인용 전제. 이유: 상태 파일 충돌 처리 복잡도
