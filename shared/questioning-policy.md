# gptaku 플러그인 질문 설계 정책 (Questioning Policy)

> 근거: 딥리서치 2회(소크라테스식 + 다양한 질문법) + A/B 시뮬레이션 3회(show-me-the-prd / kkirikkiri / vibe-sunsang / git-teacher).
> 주의: A/B는 LLM 시뮬레이션·페르소나 n=1/셀·심판 1명 → **방향성 근거**(확정 통계 아님). 단 유형마다 일관된 방향을 보임.

---

## 0. 한 줄 원칙

**추론 가능한 건 묻지 말고, 물을 땐 사용자가 진짜로 도달(니즈/통찰/이해)할 때까지 — 단 이미 도달했거나 직접 답을 원하면 즉시 멈춰라.**

질문 설계의 핵심 트레이드오프는 **깊이 ↔ 마찰**이고, 마찰의 진짜 비용은 "답변 품질"이 아니라 **이탈**이다.

---

## 1. 공통 규칙 (모든 인터뷰형 플러그인)

1. **묻지 말고 추론** — 최초 입력·맥락에서 추론 가능한 건 질문하지 않는다. 물어야 하면 "기본값 제시 + 확인"으로(예: "모바일 앱으로 가정할게요 — 아니면 말씀 주세요").
2. **핵심 먼저, 한 번에 하나** — 정보가치 가장 높은 질문을 앞에(뒤로 갈수록 답이 짧고 균일해짐=satisficing). 열린 질문은 한 번에 하나. (단순 "분류/설정 고르기"는 객관식 카드가 더 낫다.)
3. **과거 행동 > 가정** — "이 기능 쓰실래요?"(가정·X) 대신 "지금은 이 문제를 어떻게 해결하세요?"(과거 실제 행동·O). [The Mom Test]
4. **"왜" → "무엇/어떻게"** — "왜 그랬어요?"는 방어를 유발. "거기서 무엇을 하려던 거였어요?"로.
5. **답을 되짚어라(반영)** — 다음 질문 전에 직전 답을 한 문장으로 반영. 신뢰↑·마찰↓. [OARS]

## 2. ★ 종료 규칙 — 가장 중요하고 가장 어려운 부분

학계에도 검증된 "언제 멈출지" 형식 규칙이 없다. 우리 A/B가 준 운영 규칙:

### 2a. 조기 종료 금지 (모든 new 설계의 #1 실패 모드)
- **첫 답변에서 끊지 마라.** 사용자가 표면적·예의상·회피성으로 답하면 그걸 결론으로 채택하지 말 것.
- 사용자가 **진짜 결과**(real need / 스스로의 통찰 / 실제 이해)에 **본인 발화로 도달**할 때까지 턴을 이어간다. 에이전트가 "이해했다"고 단정하는 건 금물.
- **간결함은 "사용자 확인"에 종속된다.** 마찰 최소화가 과해지면 도달 직전에 끊긴다.

### 2b. 회피 대응
- 사용자가 deflect하면, **다음 질문은 반드시 구체/과거행동 앵커**로(또 다른 추상 질문 X). 예: "그냥 현황 파악…" → "최근에 누가 '이거 어때' 물었던 게 언제였어요?" [kkirikkiri v2 검증]

### 2c. 과잉 가드 (역방향 — 2a와 균형)
- 사용자가 **이미 스스로 진단**했거나, **요청이 이미 충분히 구체적**이거나, **직접 답/명령어를 명시적으로 원하면** → 더 캐묻지 말고 **즉시 제공**하라. 과잉 질문/코칭/교육 = 마찰 실패. [k6-clear / p4-aware / p3-pragmatist 검증]
- 구분 신호: 답이 *표면/회피* → 더 (2a/2b). 답이 *진짜 도달/직접 요청* → 멈춤(2c).

## 4. 피해야 할 실패 모드 (모두 A/B에서 관측됨)

- **조기 종료/원샷** — 가장 흔함 (전 유형). → 2a
- **회피 수용** — 표면 답을 진짜 니즈로 오인. → 2b
- **과잉 질문/코칭/교육** — 이미 준비된 사용자를 지치게. → 2c
- **유도신문·가정형 질문** — → 1.3, 1.4
- **설교(lecturing)** — 질문·유도 대신 일방 진단/조언/강의. [vibe-sunsang old, git-teacher]
