# insane-loop 플러그인 기획서

> 2026-07-06 초안. 근거: gjc deep-interview 분석 → Draft-First A/B 검증(4/4 통과) → 파이프라인 설계 세션.
> 관련: `docs/interview-redesign-2026-07-06.md`, `docs/reports/interview-ab-2026-07-06.md`

## 한 줄 정의

**아이디어부터 완성까지, gptaku 플러그인들을 미세 루프로 돌려 고도화하는 파이프라인 오케스트레이터.**

## 철학

한방에 완벽하게 만드는 게 아니라 **"만들고 → 검증하고 → 고치는" 작은 루프를 계속 돌려서** 품질을 올린다.
gjc deep-interview 분석의 결론이 그대로 확장된 것: 뒤에 검증·검토·개선 루프가 있으면 각 단계 입구에서
완벽을 요구할 필요가 없다. 단계는 "시작하기에 충분한" 수준으로 빠르게 통과시키고, 루프가 품질을 조인다.

## 문제

지금도 각 스테이지의 플러그인은 있다. 그런데:
1. **전환이 전부 수동이다** — PRD 만들고, goaljaby 부르고, 검증 결과 보고 고치는 걸 사용자가 기억해서 이어야 한다.
2. **컨텍스트가 단계 사이에서 끊긴다** — PRD의 가정 원장이 검증 단계에 안 넘어가고, 검증 실패가 개선 입력으로 안 이어진다.
3. **개선 루프 스테이지가 공백이다** — 검증 실패 → 문제정의 → 개선방향 → 개선 → 재검증을 묶는 주체가 없다.

## 핵심 설계 원칙

0. **루프의 엔진은 /goal이다 — 문서가 완성 정의(DoD)다.** ★ 최우선 원칙.
   루프는 "다음 단계를 제안하고 기다리는" 턴제 라우터가 아니라, **문서(PRD 수용 기준 + VALIDATION +
   blocking 가정 원장)로 정의된 완성에 도달할 때까지 골로서 계속 돌아가는 자율 실행**이다.
   - 골 종료 조건 = VALIDATION 전체 통과 **AND** blocking 가정 0. 이 조건이 goal-command와
     RECOVERY에 명시되어, 골이 중간에 끝나려 하면 RECOVERY 규칙이 개선 회차로 재진입시킨다.
   - 개선 회차(검증→리뷰→개선→재검증)는 골 **내부의** 반복 단계다. 골 밖 수동 루프가 아니다.
   - 사용자 개입은 명시된 게이트(골 시작 승인 = goaljaby Step 9, 3연속 실패 시 방향 질문)에서만.
1. **오케스트레이션만, 재구현 금지.** 하위 플러그인·CLI는 **있는 그대로 호출**한다. 어댑터 레이어,
   출력 계약 강제, 포크 없음. insane-loop가 하는 일은 ⑴ 다음 스테이지 라우팅 ⑵ 컨텍스트(파일 경로
   + 프롬프트 문맥) 전달 ⑶ 회차 기록뿐이다. 하위 플러그인이 진화해도 서로 안 깨진다.
2. **가정 원장이 관통 인터페이스.** PRD의 `## 가정 원장` → goaljaby 호출 시 "이 원장을 VALIDATION
   항목에 반영하라"는 컨텍스트로 전달 → 검증 실패·잔여 가정이 개선 루프의 문제정의 입력이 된다.
   형식 계약이 아니라 **파일 경로 + 프롬프트 문맥 전달**(느슨한 결합).
3. **리뷰어는 선택형 4종, 녹이지 않는다.** 셀프리뷰(기본, 비용 제로 — 미세 루프는 싸야 자주 돈다) /
   insane-review(GPT Pro 구독자의 깊은 회차 — ChatGPT 프로젝트 그룹핑으로 회차 간 맥락 누적) /
   codex / gjc (CLI 감지 시에만 노출, kkirikkiri check-env 패턴). 회차마다 바꿀 수 있다.
4. **승인 게이트 존중.** goaljaby Step 9(사람 검토) 등 하위 플러그인의 게이트를 절대 우회하지 않는다.
   루프의 "개선 적용"도 diff 요약 후 진행한다 — 자동 폭주 금지.

## 진입점 (어디서든 합류)

`/insane-loop [아이디어 | PRD 경로 | 프로젝트 경로]` — **현재 상태를 감지해 맞는 스테이지부터 시작**:

- 아무것도 없음 → show-me-the-prd부터
- `PRD/` 존재 → goaljaby부터 ("PRD가 이미 있네요 — 골 셋업으로 갈까요?")
- VALIDATION.md 존재·실행 흔적 있음 → 검증/개선 루프부터
- 진행 중 루프 저널 존재 → 이어서 (resume)

## 루프 회차 정의 (골 내부 반복)

한 회차 = **검증 → 문제정의·개선방향(리뷰어) → 개선 적용 → 재검증**. 이 회차는 /goal 자율 주행
**안에서** 반복된다 — 골 밖에서 사용자가 턴마다 돌리는 수동 루프가 아니다.

- **회차 시작**: VALIDATION.md 항목 실행 + 잔여 가정 원장 대조 → 실패/미확정 목록 작성
- **리뷰어 호출**: 실패 목록 + 원장 + 관련 코드를 컨텍스트로. **리뷰어 선택은 골 시작 전에 확정해
  LOOP.md에 기록**(자율 주행 중 카드 UI 상호작용 불가). 골 내 기본 = 셀프리뷰(즉시·비용 제로),
  딥 리뷰어(insane-review/codex/gjc)는 사전 설정된 시점(마일스톤 또는 리커버리 사다리 2단)에 투입
- **개선 적용**: 리뷰 결과에서 개선 태스크를 추려 적용, PROGRESS.md 갱신
- **종료 조건 (= 골 종료 조건)**: 검증 전체 통과 + 잔여 blocking 가정 0 → 완료 보고. 또는 사용자 중단.
- **회차 기록**: `LOOP.md` 저널에 회차별 한 블록(검증 결과 / 리뷰어 / 적용 태스크 / 판정 / 갭 추세).
  다음 회차와 resume의 컨텍스트가 된다. `.gjc`식 상태 디렉토리 만들지 않음 — 저널 파일 하나.

### 갭 리커버리 사다리 ★

목표(문서 DoD)와 현재 상태의 **갭이 벌어질 때**(같은 항목 반복 실패, 실패 수 증가 추세, 새 blocking
가정 발견, PRD-구현 드리프트) 단순 재시도로 무한 루프에 빠지지 않도록, RECOVERY에 단계적 대응을 박는다:

| 단계 | 트리거 | 대응 |
|---|---|---|
| 1 재시도 | 단발 실패 | 같은 접근으로 수정 후 재검증 |
| 2 접근 변경 | 같은 항목 2회차 연속 실패 | 딥 리뷰어 투입(설정된 것) 또는 다른 각도 셀프리뷰 → 접근 자체를 바꿔 재시도 |
| 3 재계획 | 3회차 연속 실패 or 구조적 갭(PRD 가정이 틀렸음이 드러남) | PLAN 갱신·PRD 가정 수정이 필요 — **사용자 게이트**로 방향 질문 후 재진입 |

gjc식 수치 게이트·점수 수학은 도입하지 않는다(A/B에서 기각). 사다리는 횟수 기반의 단순한 규칙이다.

### 골잡이 세팅이 루프의 성패를 좌우한다 ★

위 전부(DoD 종료 조건, 재진입 규칙, 리커버리 사다리, 리뷰어 설정)가 **골 시작 전에 goaljaby가 만드는
문서 5종에 새겨져야** 골이 자율 주행 중에 조기 종료·무한 재시도로 빠지지 않는다. 따라서 insane-loop의
goaljaby 스테이지는 단순 호출이 아니라 **루프형 골 세팅 컨텍스트를 전달하는 단계다**:
- 가정 원장 → VALIDATION 항목 반영 요청
- 골 종료 조건(검증 전체 통과 + blocking 가정 0) → goal-command·VALIDATION에 명시 요청
- 리커버리 사다리 + 개선 회차 재진입 규칙 → RECOVERY에 반영 요청
- 리뷰어 설정 → LOOP.md 기록
전달 방식은 원칙 1 그대로 프롬프트 컨텍스트(느슨한 결합) — goaljaby 수정 없이 Phase 1부터 가능하다.

### 드리븐 가드 — 멈췄는데 골 미달성이면 다시 건다 ★

몇 회차 돌다가 문제가 생기면 골이 그냥 멈춰버리는 경우가 실제로 발생한다. 이걸 메꾸는 3겹 가드:

**1겹 — Stop 훅 (세션 내 조기 종료 차단).** insane-loop 골이 활성(LOOP.md에 active 골 + DoD 미충족)인데
어시스턴트가 턴을 끝내려 하면 Stop 훅이 차단하고 "DoD 미달 — 리커버리 사다리 {현재 단계}로 재진입"
지시를 주입한다. 판정은 기계적(LOOP.md 상태 + VALIDATION 통과 마커) — LLM 판단에 안 맡긴다.
gjc가 Stop 훅으로 스펙 미저장 종료를 막는 것과 동일 패턴. 단 **정당한 멈춤은 통과**시킨다:
사다리 3단 사용자 게이트 대기(LOOP.md에 `gate_pending` 마커) / 사용자 명시 중단 / DoD 충족.

**2겹 — 세션 밖 재드라이브.** 세션이 아예 죽은 경우(컨텍스트 소진·크래시·이탈)는 훅이 못 잡는다:
- 기본: `/insane-loop` 재진입 시 LOOP.md resume이 멈춘 회차부터 이어감 (저널 = 재진입 지점)
- 옵트인: 네이티브 /loop 또는 Cron 스케줄로 주기 재드라이브("LOOP.md 확인 → 미완이면 재진입")

**3겹 — 멈춤 원인 자체를 리커버리로 매핑.** "몇 번 시도하다 멈추는" 전형 원인을 RECOVERY에 명시:
| 멈춤 원인 | 처리 |
|---|---|
| 같은 에러 반복 → 포기 | 사다리 2단(접근 변경·딥 리뷰어)으로 강제 전환, 포기 금지 |
| 도구·권한·환경 실패 | 실패를 LOOP.md에 기록하고 우회 경로 시도, 우회 불가면 gate_pending |
| 모호해서 물어보고 싶어짐 | 자율 주행 중엔 기본값 채택 + 가정 원장 기록 후 계속 (질문은 게이트에서만) |
| "완료했다" 착각 종료 | Stop 훅이 VALIDATION 통과 증거를 요구 — 증거 없으면 재진입 |

## 하지 않는 것 (Non-Goals)

- 하위 플러그인 로직 재구현·포크·수정 (오케스트레이션만)
- 리뷰어 출력의 형식 계약/파싱 강제 (프롬프트 컨텍스트로만 전달·수령)
- gjc식 수치 게이팅·페르소나 패널·상태 머신 (루프 저널 하나로 충분)
- 병렬 실행 오케스트레이션 (pumasi/kkirikkiri 영역 — 필요하면 그쪽을 호출하는 스테이지로)

## 구현 Phase

| Phase | 내용 | 산출 |
|---|---|---|
| 1 (MVP) | 상태 감지 라우터 + show-me-the-prd→goaljaby 브릿지(원장+루프형 골 세팅 컨텍스트 전달) + LOOP.md 저널 | 파이프라인이 손 안 대고 이어짐 |
| 2 | 골 내부 개선 루프(검증→셀프리뷰→적용→재검증) + 리커버리 사다리 + **Stop 훅 드리븐 가드** | DoD까지 멈추지 않는 루프 |
| 3 | 리뷰어 선택 4종(감지+메뉴, 골 시작 전 확정) + resume + 옵트인 주기 재드라이브 | 선택형·재드라이브 완성 |
| 4 | goaljaby에 원장→VALIDATION 반영 요청(goaljaby minor 업데이트, 별도 릴리즈) | 원장 관통 완성 |

Phase 1~3은 insane-loop 단독으로 가능(컨텍스트 전달은 프롬프트라서 하위 플러그인 무수정).
Phase 4만 goaljaby 쪽 한 줄 변경이 필요하다.
