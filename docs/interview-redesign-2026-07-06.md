## 1. 배경: gjc deep interview 분석 요약

gjc의 deep-interview(Ouroboros 포크)는 "실행 전 완전한 명세" 철학으로, 모호도를 수치 채점해
임계값(기본 0.05 = 명료도 95%) 아래로 내려갈 때까지 실행을 거부한다.

**사용자 상호작용 산수** (모호한 아이디어 1건 기준):

| 단계 | 응답 횟수 |
|------|-----------|
| Round 0 토폴로지 확인 | 1 |
| 인터뷰 라운드 (95% 도달까지) | 5~15 |
| 자유 답변마다 Refine 재확인 | +1~5 |
| Restate 종결 게이트 (최대 2루프) | 1~2 |
| 실행 브리지 승인 | 1 |
| **합계** | **10~20회 순차 응답** |

매 라운드 사이에 opus 채점 호출 + 온톨로지 추출, 마일스톤 전환마다 페르소나 패널
서브에이전트 3~4개가 낀다. 사용자는 전 과정 동안 터미널에 묶이고, 결과물은 끝까지 안 보인다.

**근본 원인 2가지:**
1. **철학의 비용** — "100% 알기 전엔 안 쓴다"는 전제는 수정 비용이 큰 자율 코딩엔 맞지만,
   초안 수정이 싼 문서/PRD엔 과잉.
2. **의사정밀성** — 모호도 점수는 LLM이 자기 이해도를 자기 채점한 숫자. 0.05라는 소수점
   임계값의 연출 비용(매 라운드 채점 호출)을 사용자가 대기 시간으로 지불.

**gjc에서 차용 / 기각:**

| 차용 (경량화) | 기각 |
|---|---|
| 적합성 게이트 — 명확한 요청은 인터뷰 생략 (≒ 우리 §2c) | 수학적 모호도 채점 · 0.05 임계값 · 매 라운드 채점 호출 |
| **"사실 vs 결정" 규칙** — 조사 가능한 사실은 묻지 않는다 | Round 0 토폴로지 게이트 · 온톨로지 스냅샷/안정성 비율 |
| 가정 원장 (established_facts의 경량판) | 페르소나 패널 (라운드당 서브에이전트 3~4) |
| 조기 종료 존중 ("그냥 해줘" → 경고 1줄 + 즉시 진행) | Refine 게이트 (답변마다 재확인 = 상호작용 2배) |
| 약한 축 겨냥의 정신 — 점수 없이 판단만 | Closure + Restate 이중 종결 게이트 · 100라운드 상한 |

**핵심 통찰:** 초안이 가장 좋은 질문 생성기다. 백지 질문은 탐색적(넓고 얕음)이라 라운드가
많이 필요하지만, 초안을 쓰다 나온 질문은 진단적(좁고 깊음)이다. 그리고 사용자에게
"추상 질문에 답하기"보다 "구체 초안의 가정 교정하기"가 인지적으로 훨씬 싸다.

단, "궁금할 때마다 그때그때 질문"을 무제한 허용하면 작성 내내 사용자를 붙잡는 또 다른
답답함이 된다 → **배칭**으로 해결. Claude Code AskUserQuestion은 1콜에 4문항까지 담을 수
있으므로 gjc의 "1라운드 1질문" 원칙은 우리 환경에서 불필요.

## 2. questioning-policy.md 개정안 (§6 신설)

기존 §0~§5는 유지. §2a(조기 종료 금지)와의 관계: §2a는 **진짜 문제 발굴 단계**(니즈/통찰)에
적용되고, §6은 그 이후의 **사양 확정 단계**에 적용된다. 문제 발굴은 깊게, 사양 확정은 초안으로.

아래 블록을 §5 뒤에 추가:

```markdown
## 6. ★ 선초안 원칙 (Draft-First) — 산출물이 문서/계획인 플러그인

> 근거: gjc(gajae-code) deep-interview 분석(2026-07-06, docs/interview-redesign-2026-07-06.md).
> 선인터뷰 10~20라운드 모델의 UX 실패에서 역설계. 초안이 가장 좋은 질문 생성기다.

### 6a. 초안이 질문보다 먼저
핵심 문제(§2a)가 잡히면, 남은 사양을 질문으로 채우지 말고 **즉시 초안을 쓴다.**
사양 질문은 초안 작성 전 1콜(최대 4문항)로 방향을 가르는 결정만 배치한다.

### 6b. 사실 vs 결정 (질문 자격 심사)
- **사실**(현재 스택, 버전, 기존 패턴, 외부 API 제약, 시장 관행) → 묻지 않는다.
  조사(코드/웹)해서 "~로 확인됨, 근거: X"로 인용 확인만.
- **결정**(목표, 범위, 트레이드오프, 원하는 동작) → 사용자에게. 애매하면 결정으로 취급.

### 6c. 가정 원장 (assumption ledger)
작성 중 만난 미확정 지점은 둘로 분류:
- **blocking** (산출물의 구조를 갈라놓음) → 질문 큐에 적립
- **non-blocking** → 합리적 기본값 채택 + 원장에 기록 (문서 내 표시 + 목록화)

### 6d. 체크포인트 배칭
큐가 쌓이면 자연스러운 중간 지점(스켈레톤 완성)에서 1콜로 배치 질문 — **최대 1회.**
작성 도중 산발적으로 질문을 던지지 않는다 (사용자를 터미널에 묶는 것 자체가 마찰).

### 6e. 납품 시 일괄 확인
초안과 함께 "확인 필요한 가정 N개"를 제시하고 한 번에 교정받아 반영한다.
인터뷰 10라운드보다 초안 보고 가정 5개 고치는 쪽이 빠르고 정확하다.

### 6f. 예산
납품 전 질문 총량: **열린 탐침 1회(§2a용) + AskUserQuestion 2콜(콜당 ≤4문항).**
초과가 필요하면 그건 질문 설계 실패다 — 기본값+원장으로 돌려라.
예외: 사용자가 명시적으로 깊은 인터뷰를 요청한 경우(--deep 옵트인)만.
```

§5 처분표 변경: show-me-the-prd 행에 "§6 선초안 재설계 대상(1순위)" 추가.
goaljaby(elicitation 상속)도 후속 적용 후보.

## 3. show-me-the-prd 커맨드 재설계안

### 현행 (v0.5.x): 선인터뷰 5~6회 직렬

```
Turn 1 열린질문 → Turn 2 기능선택 → Turn 3 데이터모델 확인
→ Turn 4 Phase 확인 → Turn 5 스택선택 → Turn 6 문서 생성
```
사용자 응답 5~6회가 전부 문서 생성 **앞에** 직렬로 배치. 마지막까지 산출물이 안 보인다.
(Turn 3/4의 preview 확인은 "구체물에 반응"이라는 좋은 본능이지만, 직렬이라 대기가 누적.)

### 재설계: 응답 3회, 두 번째 응답 직후 초안 등장

```
Turn 1  열린 탐침 (현행 유지 — Mom Test, §2a 조기종료 금지 그대로)
        ↓  (백그라운드: 기능/스택 리서치 — 사실은 조사, 묻지 않음 §6b)
Turn 2  방향 결정 1콜 (AskUserQuestion, 최대 4문항 배치):
        ① MVP 범위 (최소/중간/풀 — 리서치 기반 조합 3개)
        ② 기술 방향 (스택 후보 preview 비교표 — 리서치 기반)
        ③ 로그인 방식
        ④ 플랫폼 확인 (추론 결과 + 기본값 확인형)
        ↓
Turn 3  즉시 4종 문서 초안 Write (PRD/01~04 + README)
        - 데이터 모델·Phase 분리는 질문하지 않고 초안에 반영 (기존 Turn 3·4 제거)
        - 미확정 지점은 기본값 채택 + 각 문서에 `> ⚠️ 가정:` 블록으로 표시
        - blocking 결정이 생기면 큐 적립 (대부분의 PRD에선 발생 안 함)
        ↓
Turn 4  가정 일괄 확인 1콜 (AskUserQuestion ≤4문항, preview 활용):
        - 데이터 모델 관계도 preview + "좋아요/수정" (기존 Turn 3의 확인이 여기로 이동)
        - Phase 분리 preview + "좋아요/조정" (기존 Turn 4가 여기로 이동)
        - 가정 원장 중 임팩트 큰 것 1~2개
        ↓
Turn 5  교정 반영 → 완성도 X/10 + 가정 원장 잔여 목록 + 다음 단계 안내
```

**바뀌는 것:** 응답 5~6회 → 3회(탐침 1 + 카드 2). 산출물이 두 번째 응답 직후 등장.
데이터모델/Phase는 "질문 → 생성"에서 "생성 → 교정"으로 반전.
**안 바뀌는 것:** Turn 1의 Mom Test 열린 탐침과 §2a 가드(진짜 문제 발굴은 여전히 깊게),
preview 필드 활용, 리서치 결과를 텍스트 설명 없이 옵션에 녹이는 규칙, Step 0 setup.
